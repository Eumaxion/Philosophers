1. Thread (explicação simples)

Uma thread é como um trabalhador executando uma tarefa ao mesmo tempo que outros.

Sem threads

O programa faz tudo um de cada vez:

filosofo1 come
filosofo1 dorme
filosofo1 pensa
filosofo2 come
filosofo2 dorme

Isso não simula concorrência real.

Com threads

Agora todos os filósofos existem ao mesmo tempo.

filosofo1 pensando
filosofo3 comendo
filosofo2 pegando garfo
filosofo4 dormindo

Todos executam em paralelo.

No C usamos:

pthread_create()

Exemplo simples:

void *routine(void *arg)
{
    printf("Hello from thread\n");
    return NULL;
}

int main()
{
    pthread_t t;

    pthread_create(&t, NULL, routine, NULL);
    pthread_join(t, NULL);
}
2. Recurso compartilhado

No philosophers temos:

filósofos
garfos

Os garfos são compartilhados.

Exemplo:

P1 --- fork --- P2

Então dois filósofos podem tentar pegar o mesmo garfo.

Isso cria problemas.

3. Race Condition (corrida)

Race condition acontece quando duas threads acessam o mesmo dado ao mesmo tempo.

Exemplo:

fork disponível

Thread 1 lê:

fork = livre

Thread 2 lê:

fork = livre

As duas pegam o garfo.

💥 Problema: duas threads usando o mesmo recurso.

4. Mutex (cadeado)

Um mutex é um cadeado.

Só uma thread pode entrar.

Exemplo visual:

garfo 🔒

Fluxo:

Thread tenta pegar garfo
        │
        ▼
lock(mutex)
        │
        ▼
usa garfo
        │
        ▼
unlock(mutex)

Em C:

pthread_mutex_lock(&fork);

e depois:

pthread_mutex_unlock(&fork);
5. Deadlock (travamento)

Deadlock é quando todos ficam esperando para sempre.

Exemplo clássico:

P1 pega fork esquerdo
P2 pega fork esquerdo
P3 pega fork esquerdo
P4 pega fork esquerdo
P5 pega fork esquerdo

Agora todos querem o fork direito.

Mas:

cada fork direito está na mão do vizinho

Então:

P1 espera P2
P2 espera P3
P3 espera P4
P4 espera P5
P5 espera P1

🔒 Todos travam para sempre.

Isso é deadlock.

6. Starvation (fome infinita)

Starvation acontece quando um filósofo nunca consegue comer.

Exemplo:

P1 sempre pega garfos primeiro
P2 sempre perde

Resultado:

P2 nunca come
P2 morre

1. Ideia geral do projeto

O Dining Philosophers Problem é um problema clássico de concorrência.

Temos:

N filósofos

N garfos

Cada filósofo precisa de 2 garfos para comer

Estados possíveis:

🍝 Eating

😴 Sleeping

🤔 Thinking

Problemas que precisamos evitar:

Race conditions

Deadlocks

Starvation

No teu código:

Cada filósofo = uma thread

Cada garfo = um mutex

Existe uma thread extra chamada monitor

2. Estrutura principal do programa

Existem 2 estruturas principais:

t_table  → dados globais da simulação
t_philo  → dados de cada filósofo
3. Estrutura t_philo
typedef struct s_philo
{
    int id;
    pthread_t thread;

Cada filósofo tem:

um id

uma thread

Controle da última refeição
long last_meal_ms;

Guarda:

última vez que o filósofo comeu

O mutex protege contra:

monitor lendo enquanto filósofo escreve
Controle de refeições
int meals_eaten;
t_mutex meals_eaten_mutex;

Serve para:

verificar se já comeu o número obrigatório
Garfo esquerdo

t_mutex l_fork_mutex;

Cada filósofo possui seu garfo esquerdo.

Garfo direito
bool *right_fork;
t_mutex *r_fork_mutex;

O garfo direito é o garfo esquerdo do próximo filósofo.

Por isso é ponteiro.

Exemplo com 5 filósofos:

P1: left fork = F1
P1: right fork = F2

P2: left fork = F2
P2: right fork = F3
Ponteiro para a mesa
t_table *table;

Isso permite que o filósofo acesse:

tempos

número de filósofos

flags globais
